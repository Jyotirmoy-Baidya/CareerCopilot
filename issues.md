### ISSUE-010 — `logout` does not revoke the refresh token in Redis (security gap)
**Severity:** HIGH  
**Status:** Open  
**File:** `services/auth/src/routes/logout.ts`  
**Detail:**  
The plan requires `POST /auth/logout — delete refresh token from Redis`. The current implementation only calls `res.clearCookie('refreshToken')`. This clears the cookie on the client, but:
1. `login.ts` never stores the refresh token in Redis, so there is nothing to revoke server-side.
2. `logout.ts` never touches Redis at all.
3. `refresh.ts` never validates the refresh token against Redis.

Result: after logout, anyone who captured the refresh token value (browser devtools, server logs, XSS) can still obtain new access tokens for up to 30 days. Logout is effectively cosmetic only.  
**Fix:** On login, `SET refresh:{userId} {token} EX 2592000` in Redis. On logout, `DEL refresh:{userId}`. In refresh route, check `redis.get(refresh:{userId})` equals the token before issuing a new one. Auth service already has `ioredis` as a dependency — it just needs to be wired in.

**Severity:** MEDIUM

### ISSUE-012 — Collaboration viewer write-drop blocks the WRONG Yjs message type (security bug)
**Severity:** HIGH  
**Status:** Open  
**File:** `services/collaboration/src/roles.ts` → `isYjsUpdateMessage`, `services/collaboration/src/index.ts`  
**Detail:**  
`isYjsUpdateMessage` returns `true` when `data[0] === 1`. In the y-websocket protocol, the top-level message type byte means:
- `0` = messageSync (includes all document updates)
- `1` = messageAwareness (cursor positions, presence)
- `2` = messageAuth

The current code blocks **awareness messages** (cursor movements) from viewers, NOT document updates. Viewers can still push actual document edits because those arrive as `[0 (messageSync), 2 (update), ...]`. The viewer write restriction is bypassed entirely.  
**Fix:** Change `isYjsUpdateMessage` to check for sync update messages:
```ts
// Block Yjs document updates: messageSync(0) + messageYjsUpdate(2)
return data.length > 1 && data[0] === 0 && data[1] === 2;
```
Awareness messages (`data[0] === 1`) should be allowed through — viewers should still be able to show their cursor.

### ISSUE-013 — `sync/flush` `roadmap_node_update` has no user ownership check (IDOR)
**Severity:** HIGH  
**Status:** Open  
**File:** `services/sync/src/routes/flush.ts` line 76  
**Detail:**  
The `roadmap_node_update` operation updates a roadmap node using only its ID:
```ts
.where(eq(roadmapNodes.id, op.payload.nodeId))  // ← no userId check
```
The `task_update` operation (same file, line 72) correctly guards with:
```ts
.where(and(eq(dailyTasks.id, op.payload.taskId), eq(dailyTasks.userId, userId)))
```
Any authenticated user can mark any other user's roadmap node as complete by knowing (or guessing) its UUID. UUIDs are not secret — they appear in API responses.  
**Fix:** Join through `roadmaps` to verify ownership:
```ts
.where(and(
  eq(roadmapNodes.id, op.payload.nodeId),
  eq(roadmaps.userId, userId)  // requires join or subquery
))
```
Or: fetch the node first, verify `roadmap.userId === userId`, then update.

### ISSUE-014 — NextAuth stores a 15-min access token with no refresh mechanism (session breaks silently)
**Severity:** MEDIUM  
**Status:** Open  
**File:** `apps/web/src/lib/auth.ts`  
**Detail:**  
The `authorize` callback stores the service-level `accessToken` (15-min JWT) in the NextAuth session. After 15 minutes the token expires, but NextAuth's own session remains valid — it only checks its own `NEXTAUTH_SECRET`, not the embedded token's expiry. All server-component `fetch` calls in `api.ts` then receive 401s from the microservices, silently returning `null` or `[]`. Users see a broken dashboard with no error message and no redirect to login.  
**Fix:** Two options:
1. (Simple) Increase auth service access token TTL to 7 days to match NextAuth session duration.
2. (Proper) In the NextAuth `jwt` callback, check `token.exp` against `Date.now()`. If expired, call the auth service `/auth/refresh` endpoint using the stored refresh cookie to rotate tokens silently. Return the new `accessToken`; on failure, set `token.error = 'RefreshTokenExpired'` and handle in the session callback to force sign-out.

### ISSUE-016 — `SyncdocEditor` passes a `useRef` value to `useEditor` — collaboration doc never connects
**Severity:** HIGH  
**Status:** Open  
**File:** `apps/web/src/components/client/editor/SyncdocEditor.tsx` lines 46–68  
**Detail:**  
The Yjs providers are created in a `useEffect` and stored in `providersRef` (a `useRef`). The `useEditor` call then does:
```ts
Collaboration.configure({ document: providersRef.current?.ydoc ?? new Y.Doc() })
// dependency: [providersRef.current?.ydoc]
```
Two compounding problems:
1. On first render the `useEffect` has not run yet, so `providersRef.current` is `null`. The Collaboration extension gets a fresh disconnected `new Y.Doc()`.
2. `useRef` mutations do **not** trigger re-renders. When the effect later sets `providersRef.current`, React does not re-run `useEditor`, so the editor keeps the orphaned YDoc forever. Real-time collaboration is completely non-functional.

**Fix:** Replace `useRef<YjsProviders>` with `useState<YjsProviders | null>`. State updates trigger re-renders, so `useEditor` will reinitialise with the real connected ydoc:
```ts
const [providers, setProviders] = useState<YjsProviders | null>(null);
// in useEffect: setProviders(createYjsProviders(...))
// in useEditor dep array: [providers?.ydoc]
```

### ISSUE-017 — `onboard.ts` parses `knownSlugs` but never saves them — gap analysis is always wrong
**Severity:** HIGH  
**Status:** Open  
**File:** `services/auth/src/routes/onboard.ts` line 22  
**Detail:**  
The onboarding route accepts `knownSlugs: string[]` via Zod, but the destructure on line 22 only pulls `targetRole` and `weeklyGoalHrs`:
```ts
const { targetRole, weeklyGoalHrs } = parsed.data;  // knownSlugs silently dropped
```
`knownSlugs` is never used. The `userSkills` table import at line 4 is dead code. The recommendation engine's `gapAnalysis()` queries `userSkills` to find what the user already knows — since nothing is ever inserted, it always sees an empty skill set and generates a full roadmap from zero. The "skip what you know" feature is completely non-functional.  
**Fix:**
```ts
const { targetRole, knownSlugs, weeklyGoalHrs } = parsed.data;
// after updating the user:
if (knownSlugs.length > 0) {
  const slugToId = await db.query.skillNodes.findMany({
    where: (s, { inArray }) => inArray(s.slug, knownSlugs),
    columns: { id: true },
  });
  await db.insert(userSkills).values(
    slugToId.map(s => ({ userId, skillId: s.id }))
  ).onConflictDoNothing();
}
```
